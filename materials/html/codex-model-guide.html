Picking a GPT-5 model for coding work

Do not turn model choice into theater. Pick the smallest model that keeps the loop moving, then move up only when the task is genuinely ambiguous, high-risk, or review-heavy.

Good default

Start with the normal default your team already trusts. Most work does not fail because the model was one tier too small. It fails because the repo context, constraints, or verification path were weak.

Move up when

  • the problem is ambiguous and the agent needs stronger judgment
  • the change spans multiple files or trust boundaries
  • you are asking for review, debugging, or architectural reasoning
  • the cost of a wrong answer is higher than the extra latency

Move down when

  • you are iterating on a narrow implementation slice
  • you already know the exact files and acceptance criteria
  • you want a fast loop for formatting, cleanup, or small edits

What matters more than the picker

  • a short AGENTS.md with context and constraints
  • one executable check before you trust the output
  • a prompt that names the goal and the done criteria clearly
  • one review pass when the slice gets bigger

Shortcut

If you are not sure: keep the current model, improve the task framing, and only then reconsider the picker.
